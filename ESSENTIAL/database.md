## 1. Naming Conventions

All database objects follow a strict `snake_case` naming policy. No exceptions. Deviations require a team lead decision and must be documented in the migration comment.

### 1.1 Tables

| Rule | Example |
|------|---------|
| `snake_case`, always plural | `trade_agreements`, `hs_codes`, `calculation_executions` |
| No abbreviations unless universally accepted | `hs_codes` (HS is a domain term), not `trd_agr` |
| No table prefix (no `tbl_`) | `countries`, not `tbl_countries` |

### 1.2 Columns

| Rule | Example |
|------|---------|
| `snake_case` | `agreement_name`, `effective_date`, `is_deleted` |
| Booleans prefixed with `is_` or `has_` | `is_deleted`, `is_eu_member`, `is_originating`, `is_enabled` |
| Timestamps always suffixed with `_at` | `created_at`, `executed_at`, `deleted_at` |
| Actor columns suffixed with `_by` | `created_by`, `updated_by`, `deleted_by` |
| Foreign key columns: `{referenced_singular}_id` | `country_id`, `hs_code_id`, `agreement_id`, `product_id` |

### 1.3 Constraint Naming

All constraint names are explicit. PostgreSQL auto-generated names are never acceptable in production migrations.

| Constraint Type | Pattern | Example |
|----------------|---------|---------|
| Primary key | `pk_{table}` | `pk_trade_agreements` |
| Foreign key | `fk_{table}_{referenced_table}` | `fk_product_rules_hs_codes` |
| Unique | `uq_{table}_{column}` | `uq_countries_iso_code` |
| Check | `ck_{table}_{column}` | `ck_materials_currency` |
| Index | `ix_{table}_{column(s)}` | `ix_trade_agreements_country_id` |
| Unique index | `uix_{table}_{column}` | `uix_rule_definitions_rule_code` |
| Partial index | `pix_{table}_{column}_{qualifier}` | `pix_trade_agreements_status_active` |

### 1.4 Sequences and Other Objects

| Object | Pattern | Example |
|--------|---------|---------|
| View | `vw_{name}` | `vw_active_trade_agreements` |
| Function | `fn_{name}` | `fn_calculate_origination_percentage` |
| Trigger | `trg_{table}_{event}` | `trg_trade_agreements_before_update` |
| Schema | lowercase single word | `public`, `audit`, `lookup`, `rule_engine` |

## 2. Schema Organization

The database is organized across four schemas. Each schema has a distinct purpose and access role.

| Schema | Purpose | Contains |
|--------|---------|---------|
| `public` | Core business entities | `trade_agreements`, `countries`, `hs_codes`, `product_rules`, `materials`, `finished_products`, `calculation_executions` |
| `audit` | Immutable audit trail | Append-only audit log tables, partitioned by year |
| `lookup` | Reference / static data | Currency codes, region lists, enumeration tables |
| `rule_engine` | Rule metadata | `rule_definitions`, rule version history, expression cache |

### Schema Assignment in EF Core

```csharp
// In DbContext or model builder
modelBuilder.HasDefaultSchema("public");

// Override per entity when needed
builder.ToTable("rule_definitions", schema: "rule_engine");
```

The application role `app_user` has `USAGE` on `public` and `rule_engine` schemas, `SELECT` only on `lookup`, and no direct access to `audit` (writes go through a dedicated function).

## 3. Primary Keys

### 3.1 UUID v7 (Sequential UUIDs)

Every business table uses a `UUID` primary key. UUID v7 is preferred over UUID v4 and over `SERIAL`/`BIGSERIAL`.

**Rationale:**

- UUID v7 is time-ordered (first 48 bits are a millisecond timestamp), which gives B-tree index locality similar to sequential integers while remaining globally unique across distributed systems.
- UUID v4 causes index fragmentation — each insert goes to a random B-tree page, causing page splits and bloat.
- `SERIAL` creates coupling between database sequences and application logic; it also leaks record counts.
- UUIDs can be generated by the application before the database round-trip, enabling batch inserts and offline ID assignment.

### 3.2 Generation Strategy

Generate UUID v7 in the application layer using `UuidNext` or equivalent. Do not rely on PostgreSQL `gen_random_uuid()` (which produces v4).

```csharp
// Use UuidNext NuGet package
public static Guid NewId() => Uuid.NewDatabaseFriendly(Database.PostgreSql);
```

### 3.3 Column Definition

```sql
id UUID NOT NULL DEFAULT gen_random_uuid(),
CONSTRAINT pk_trade_agreements PRIMARY KEY (id)
```

The `DEFAULT gen_random_uuid()` is a fallback only. The application always supplies the value.

## 4. Foreign Keys

### 4.1 Always Explicit

Every foreign key relationship must have an explicit `CONSTRAINT` clause with a named constraint. Implicit FKs (EF Core without explicit naming) are never allowed in production migrations.

```sql
CONSTRAINT fk_product_rules_hs_codes
    FOREIGN KEY (hs_code_id) REFERENCES public.hs_codes (id)
```

### 4.2 Cascade Policy

| Relationship Type | Cascade Rule |
|-------------------|-------------|
| Business data parent → child | `ON DELETE RESTRICT` (default) |
| Lookup/reference data | `ON DELETE RESTRICT` |
| Soft-deletable parent | No cascade; soft delete propagated by application |
| Audit records | `ON DELETE NO ACTION` |

**Cascade delete is forbidden on all business data tables.** Deleting a parent row must never silently delete children. All deletions are soft (see Section 8). If a cascade is ever needed due to technical constraints, it requires explicit team lead approval and a written justification in the migration file.

### 4.3 Indexed Foreign Keys

Every foreign key column must have a corresponding index. PostgreSQL does not auto-index FK columns. Missing FK indexes cause sequential scans on the child table during parent table operations.

```sql
CREATE INDEX ix_trade_agreements_country_id ON public.trade_agreements (country_id);
CREATE INDEX ix_product_rules_hs_code_id    ON public.product_rules (hs_code_id);
CREATE INDEX ix_materials_origin_country_id ON public.materials (origin_country_id);
```

## 5. Indexes

### 5.1 When to Add an Index

Add an index when:
- The column appears in a `WHERE` clause on queries returning less than ~15% of rows.
- The column is a foreign key (mandatory — see Section 4.3).
- The column is used in `ORDER BY` on paginated queries.
- A composite index can serve as a covering index for a high-frequency read path.

Do not add an index speculatively. Each index has a write overhead cost. Run `EXPLAIN ANALYZE` on real query plans before adding.

### 5.2 Composite Index Column Order

For composite indexes, place columns in this order:
1. Equality predicates first (e.g., `tenant_id = @TenantId`)
2. Range predicates second (e.g., `effective_date BETWEEN ...`)
3. Sort columns last (e.g., `agreement_name` for `ORDER BY`)

```sql
-- Correct: equality (tenant_id, status) before ORDER BY column
CREATE INDEX ix_trade_agreements_tenant_status
    ON public.trade_agreements (tenant_id, status);

-- Correct: equality + range
CREATE INDEX ix_calculation_executions_tenant_executed
    ON public.calculation_executions (tenant_id, executed_at DESC);
```

### 5.3 Partial Indexes for Soft Deletes

Always create partial indexes that exclude deleted rows on high-traffic tables. This keeps the index small and avoids scanning tombstoned rows.

```sql
-- Partial index: only active (non-deleted) trade agreements
CREATE INDEX pix_trade_agreements_status_active
    ON public.trade_agreements (tenant_id, status)
    WHERE is_deleted = FALSE;

-- Partial index: only enabled rule definitions
CREATE INDEX pix_rule_definitions_enabled
    ON rule_engine.rule_definitions (agreement_id, priority, execution_order)
    WHERE is_deleted = FALSE AND is_enabled = TRUE;
```

### 5.4 Covering Indexes

For Dapper read queries that always fetch the same column set, a covering index can eliminate heap fetches entirely.

```sql
-- Covering index for trade agreement list queries
CREATE INDEX ix_trade_agreements_covering
    ON public.trade_agreements (tenant_id, status)
    INCLUDE (agreement_name, effective_date, expiry_date, country_id)
    WHERE is_deleted = FALSE;
```

### 5.5 GIN Indexes for JSONB

JSONB columns used in `WHERE` clauses require GIN indexes (see Section 9.3).

## 6. Constraints

### 6.1 NOT NULL by Default

Every column is `NOT NULL` unless there is an explicit, documented reason for nullability. Nullable columns must have a comment explaining the null semantics.

```sql
-- Correct
agreement_name  VARCHAR(200) NOT NULL,
updated_at      TIMESTAMPTZ,         -- NULL until first update
deleted_at      TIMESTAMPTZ,         -- NULL until soft-deleted
```

### 6.2 CHECK Constraints for Enumerations

Database-level CHECK constraints enforce domain integrity for status and enumeration columns. Do not rely solely on application-layer validation.

```sql
CONSTRAINT ck_trade_agreements_status
    CHECK (status IN ('Draft', 'Active', 'Expired', 'Suspended')),

CONSTRAINT ck_materials_currency
    CHECK (currency ~ '^[A-Z]{3}$'),

CONSTRAINT ck_product_rules_rule_type
    CHECK (rule_type IN ('WO', 'CTH', 'CTSH', 'RVC', 'SP', 'Combination')),

CONSTRAINT ck_calculation_executions_decision
    CHECK (decision IN ('Originating', 'NonOriginating', 'Inconclusive'))
```

### 6.3 UNIQUE Constraints

```sql
CONSTRAINT uq_countries_iso_code       UNIQUE (iso_code),
CONSTRAINT uq_hs_codes_hs_code         UNIQUE (hs_code),
CONSTRAINT uq_rule_definitions_rule_code UNIQUE (rule_code)
```

For soft-delete tables, unique constraints must be partial to allow re-creation of logically deleted records:

```sql
-- Unique only among active (non-deleted) records
CREATE UNIQUE INDEX uix_trade_agreements_name_tenant_active
    ON public.trade_agreements (agreement_name, tenant_id)
    WHERE is_deleted = FALSE;
```

## 7. Audit Fields

Every business table carries the following nine audit columns. These are mandatory with no exceptions. The `audit` schema tables do not repeat these — they have their own structure.

```sql
created_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
created_by   VARCHAR(256) NOT NULL,
updated_at   TIMESTAMPTZ,
updated_by   VARCHAR(256),
is_deleted   BOOLEAN      NOT NULL DEFAULT FALSE,
deleted_at   TIMESTAMPTZ,
deleted_by   VARCHAR(256),
ip_address   VARCHAR(45),   -- IPv6 compatible (max 45 chars for IPv6 with port)
machine      VARCHAR(256)
```

### 7.1 Population Rules

| Field | When set | By whom |
|-------|----------|---------|
| `created_at` | On INSERT, once, never updated | Database default `NOW()` confirmed by application |
| `created_by` | On INSERT | Application — authenticated user identity (email or UPN) |
| `updated_at` | On UPDATE, never on INSERT | Application interceptor / EF Core `SaveChanges` override |
| `updated_by` | On UPDATE | Application — current user identity |
| `is_deleted` | Set to `TRUE` on soft delete | Application |
| `deleted_at` | Set on soft delete | Application — `NOW()` at soft delete time |
| `deleted_by` | Set on soft delete | Application — current user identity |
| `ip_address` | On INSERT and UPDATE | Application — from HTTP context `RemoteIpAddress` |
| `machine` | On INSERT and UPDATE | Application — server hostname or service identifier |

### 7.2 EF Core Audit Interceptor

Implement a `SaveChangesInterceptor` that populates these fields automatically. Never rely on callers to set them manually.

```csharp
public class AuditInterceptor : SaveChangesInterceptor
{
    public override InterceptionResult<int> SavingChanges(
        DbContextEventData eventData, InterceptionResult<int> result)
    {
        ApplyAuditFields(eventData.Context);
        return base.SavingChanges(eventData, result);
    }

    private static void ApplyAuditFields(DbContext context)
    {
        var entries = context.ChangeTracker.Entries<IAuditableEntity>();
        foreach (var entry in entries)
        {
            if (entry.State == EntityState.Added)
            {
                entry.Entity.CreatedAt = DateTime.UtcNow;
                // CreatedBy, IpAddress, Machine set from ICurrentUserContext
            }
            if (entry.State == EntityState.Modified)
            {
                entry.Entity.UpdatedAt = DateTime.UtcNow;
                // UpdatedBy, IpAddress, Machine set from ICurrentUserContext
            }
        }
    }
}
```

## 8. Soft Deletes

Physical `DELETE` statements are forbidden on all business tables. All removals are soft deletes that set `is_deleted = TRUE`, `deleted_at`, and `deleted_by`.

### 8.1 Application of Soft Delete

```csharp
// Application service — never call context.Remove()
entity.IsDeleted   = true;
entity.DeletedAt   = DateTime.UtcNow;
entity.DeletedBy   = _currentUser.Identity;
await _context.SaveChangesAsync();
```

### 8.2 EF Core Global Query Filter

Every entity that participates in soft delete must have a global query filter configured in its `IEntityTypeConfiguration<T>`. This ensures that `is_deleted = TRUE` rows are excluded automatically from all LINQ queries.

```csharp
builder.HasQueryFilter(x => !x.IsDeleted);
```

To query deleted records (e.g., admin restore functionality), use `IgnoreQueryFilters()`:

```csharp
var deletedAgreements = await _context.TradeAgreements
    .IgnoreQueryFilters()
    .Where(x => x.IsDeleted && x.TenantId == tenantId)
    .ToListAsync();
```

### 8.3 Partial Index on Soft Delete

Every soft-delete table must have a partial index excluding deleted rows on its most-used filter columns (see Section 5.3). This prevents bloat in the active data index.

## 9. JSONB Usage

### 9.1 When to Use JSONB

Use JSONB for:
- Rule engine parameters where schema varies by rule type (`rule_definitions.parameters`)
- Audit snapshots capturing full before/after state (`calculation_executions.rules_executed`, `rule_results`, `input_data`, `output_data`)
- Configuration blobs with optional, sparse key sets

Do not use JSONB for:
- Data that is frequently filtered or joined on individual fields — normalize it into columns
- Data with a stable, well-known schema — use typed columns
- Foreign key targets — UUIDs and relationship columns must always be first-class columns

### 9.2 JSONB Column Definition

```sql
parameters   JSONB,           -- rule_definitions: varies by rule_type
rules_executed JSONB,         -- calculation_executions: snapshot of applied rules
rule_results   JSONB,         -- calculation_executions: per-rule outcomes
input_data     JSONB,         -- calculation_executions: request snapshot
output_data    JSONB          -- calculation_executions: response snapshot
```

### 9.3 GIN Indexes for JSONB

```sql
-- GIN index for containment queries on rule parameters
CREATE INDEX ix_rule_definitions_parameters_gin
    ON rule_engine.rule_definitions USING GIN (parameters);

-- GIN index for calculation execution results
CREATE INDEX ix_calculation_executions_rule_results_gin
    ON public.calculation_executions USING GIN (rule_results);
```

### 9.4 Querying JSONB in Dapper

```csharp
// Containment operator @>
public const string GetByRuleType = @"
    SELECT id, rule_name, parameters
    FROM   rule_engine.rule_definitions
    WHERE  parameters @> @Filter::jsonb
      AND  is_deleted = FALSE";

// Usage
var filter = JsonSerializer.Serialize(new { threshold_type = "RVC" });
var results = await connection.QueryAsync<RuleDefinition>(
    RuleDefinitionQueries.GetByRuleType,
    new { Filter = filter });
```

## 10. Partitioning

### 10.1 Audit Table Partitioning

Tables in the `audit` schema are range-partitioned by year on the `occurred_at` column. This keeps query performance predictable and enables dropping old partitions without `DELETE` overhead.

```sql
CREATE TABLE audit.entity_changes (
    id           UUID        NOT NULL,
    table_name   VARCHAR(100) NOT NULL,
    operation    CHAR(1)     NOT NULL,  -- I/U/D
    occurred_at  TIMESTAMPTZ NOT NULL,
    changed_by   VARCHAR(256) NOT NULL,
    old_values   JSONB,
    new_values   JSONB
) PARTITION BY RANGE (occurred_at);

CREATE TABLE audit.entity_changes_2024
    PARTITION OF audit.entity_changes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE audit.entity_changes_2025
    PARTITION OF audit.entity_changes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
```

### 10.2 Calculation Execution History

`calculation_executions` may be partitioned by `executed_at` year once volume exceeds 10 million rows. Create the partition definition in advance; the application code does not change.

### 10.3 Retention Policy

Old partitions are detached and archived to cold storage after 7 years to comply with EU trade documentation retention requirements. Detach with `ALTER TABLE ... DETACH PARTITION`, then dump with `pg_dump --table`.

## 11. Views

### 11.1 Naming Convention

All views are prefixed with `vw_`. Views are read-only projections; no `INSTEAD OF` triggers on views.

```sql
CREATE OR REPLACE VIEW public.vw_active_trade_agreements AS
SELECT
    ta.id,
    ta.agreement_name,
    ta.effective_date,
    ta.expiry_date,
    ta.status,
    ta.tenant_id,
    c.country_name,
    c.iso_code,
    c.region
FROM  public.trade_agreements ta
INNER JOIN public.countries c ON c.id = ta.country_id
WHERE ta.is_deleted = FALSE
  AND ta.status = 'Active';
```

### 11.2 Purpose

Views serve as stable read-model contracts for complex Dapper queries. When a Dapper query requires three or more joins, consider moving the join logic to a view. Application code then queries the view, isolating it from underlying table restructuring.

Views are never used as the write target. All writes go through EF Core to tracked entity tables.

## 12. Functions

Stored functions are permitted only for:
- Complex multi-step calculations that benefit from set-based SQL (e.g., aggregating material costs across a product BOM)
- Scheduled maintenance tasks (partition management, statistics refresh)
- Trigger functions (e.g., enforcing immutability on audit rows)

Business logic that involves branching, rule evaluation, or state transitions belongs in the C# application layer, not in stored functions. Stored functions are harder to test, version, and debug than application code.

```sql
-- Acceptable: pure calculation helper
CREATE OR REPLACE FUNCTION rule_engine.fn_calculate_non_originating_percentage(
    p_product_id UUID,
    p_tenant_id  UUID
) RETURNS NUMERIC AS $$
BEGIN
    -- set-based cost aggregation
    RETURN (
        SELECT COALESCE(SUM(m.cost * m.quantity), 0)
        FROM   public.materials m
        WHERE  m.finished_product_id = p_product_id
          AND  m.is_originating = FALSE
          AND  m.is_deleted = FALSE
    );
END;
$$ LANGUAGE plpgsql STABLE;
```

## 13. Transactions

### 13.1 Unit of Work Pattern

EF Core's `DbContext` is the unit of work. A single `SaveChangesAsync()` call wraps all changes in one database transaction. Do not open manual transactions for operations that EF Core's change tracker can handle.

For cross-aggregate operations or mixed EF Core + Dapper writes, use `IDbContextTransaction`:

```csharp
await using var tx = await _context.Database.BeginTransactionAsync();
try
{
    // EF Core changes
    await _context.SaveChangesAsync();
    // Dapper writes using tx.GetDbTransaction()
    await connection.ExecuteAsync(sql, param, tx.GetDbTransaction());
    await tx.CommitAsync();
}
catch
{
    await tx.RollbackAsync();
    throw;
}
```

### 13.2 Isolation Levels

| Level | When to use |
|-------|-------------|
| `READ COMMITTED` | Default for all application queries |
| `REPEATABLE READ` | Read-modify-write on financial calculations (e.g., RVC percentage computation across materials) |
| `SERIALIZABLE` | Never in application code; reserved for batch maintenance scripts with explicit justification |

Escalate isolation only when you have demonstrated that a lower level produces incorrect results. Higher isolation levels increase lock contention and reduce throughput.

## 14. Migration Strategy

### 14.1 EF Core Migrations — Forward Only

Migrations are always forward-only. The `Down()` method in every migration must be left empty or throw `NotSupportedException`. Down migrations are not executed in any environment.

```csharp
protected override void Down(MigrationBuilder migrationBuilder)
{
    // Down migrations are not supported in production.
    // To reverse this change, create a new forward migration.
    throw new NotSupportedException("Down migrations are not supported.");
}
```

### 14.2 Migration Naming

```
{YYYYMMDDHHMMSS}_{DescriptiveName}
```

Examples:
- `20240315143022_AddTradeAgreementsTable`
- `20240612090015_AddGinIndexRuleDefinitionsParameters`
- `20250101000000_PartitionCalculationExecutionsByYear`

The timestamp is UTC. Use `dotnet ef migrations add {DescriptiveName}` — EF Core stamps the timestamp automatically.

### 14.3 Review Before Apply

Every migration SQL must be reviewed by a second developer before applying to any environment beyond local development. Generate the idempotent SQL script:

```bash
dotnet ef migrations script --idempotent --output migration-review.sql
```

Confirm the script:
- Contains no `DROP TABLE` or `DROP COLUMN` without a prior deprecation period
- Names all constraints explicitly
- Does not lock tables for longer than ~2 seconds on the expected data volume (use `CONCURRENTLY` for index creation on large tables)

### 14.4 Concurrent Index Creation

For tables with more than 100,000 rows, indexes must be created `CONCURRENTLY` to avoid locking. EF Core does not support `CONCURRENTLY` natively; use a raw SQL migration:

```csharp
migrationBuilder.Sql(
    "CREATE INDEX CONCURRENTLY IF NOT EXISTS ix_calculation_executions_tenant_executed " +
    "ON public.calculation_executions (tenant_id, executed_at DESC);");
```

## 15. Dapper Standards

### 15.1 SQL in Static String Constants

All Dapper SQL is stored in `static` classes with `const string` fields. No inline SQL strings in service or repository methods. This enables IDE navigation, grep-ability, and code review.

```csharp
public static class TradeAgreementQueries
{
    public const string GetById = @"
        SELECT ta.id, ta.agreement_name, ta.effective_date, ta.expiry_date, ta.status,
               c.country_name, c.iso_code
        FROM   trade_agreements ta
        INNER JOIN countries c ON c.id = ta.country_id
        WHERE  ta.id = @Id
          AND  ta.tenant_id = @TenantId
          AND  ta.is_deleted = FALSE";

    public const string Search = @"
        SELECT ta.id, ta.agreement_name, ta.status, c.country_name
        FROM   trade_agreements ta
        INNER JOIN countries c ON c.id = ta.country_id
        WHERE  ta.tenant_id = @TenantId
          AND  ta.is_deleted = FALSE
          AND  (@Name IS NULL OR ta.agreement_name ILIKE '%' || @Name || '%')
        ORDER BY ta.agreement_name
        LIMIT @PageSize OFFSET @Offset";
}
```

### 15.2 Multi-Table Joins with SplitOn

When a Dapper query populates a parent object and one or more child objects from a single query, use `QueryAsync` with `splitOn`:

```csharp
var result = await connection.QueryAsync<TradeAgreement, Country, TradeAgreement>(
    TradeAgreementQueries.GetById,
    (agreement, country) =>
    {
        agreement.Country = country;
        return agreement;
    },
    new { Id = id, TenantId = tenantId },
    splitOn: "country_name");
```

### 15.3 DynamicParameters for Conditional Queries

```csharp
var p = new DynamicParameters();
p.Add("TenantId", tenantId);
p.Add("PageSize", request.PageSize);
p.Add("Offset", (request.Page - 1) * request.PageSize);
p.Add("Name", request.Name);  // NULL when not provided — SQL handles with @Name IS NULL
```

### 15.4 Always Async

All Dapper calls use the `Async` variants: `QueryAsync`, `QuerySingleOrDefaultAsync`, `ExecuteAsync`, `ExecuteScalarAsync`. Synchronous Dapper methods are never used in application code.

### 15.5 Connection Management

Dapper repositories receive `IDbConnection` via constructor injection. Do not open/close connections manually inside query methods — the DI container manages connection lifetime through the registered `NpgsqlConnection` scoped lifetime.

## 16. EF Core Standards

### 16.1 IEntityTypeConfiguration per Entity

Every entity has exactly one `IEntityTypeConfiguration<T>` class in the Infrastructure layer. No `DataAnnotations` (`[Required]`, `[MaxLength]`, etc.) are placed on domain model classes.

```csharp
public class TradeAgreementConfiguration : IEntityTypeConfiguration<TradeAgreement>
{
    public void Configure(EntityTypeBuilder<TradeAgreement> builder)
    {
        builder.ToTable("trade_agreements");
        builder.HasKey(x => x.Id);

        builder.Property(x => x.AgreementName)
               .IsRequired()
               .HasMaxLength(200);

        builder.Property(x => x.Status)
               .IsRequired()
               .HasMaxLength(50)
               .HasConversion<string>();  // value converter for enum

        builder.HasIndex(x => new { x.TenantId, x.Status })
               .HasDatabaseName("ix_trade_agreements_tenant_status");

        builder.HasQueryFilter(x => !x.IsDeleted);

        // Shadow properties for audit fields not on the domain model
        builder.Property<string>("CreatedBy").IsRequired().HasMaxLength(256);
        builder.Property<DateTime>("CreatedAt").IsRequired();
        builder.Property<string?>("UpdatedBy").HasMaxLength(256);
        builder.Property<DateTime?>("UpdatedAt");
        builder.Property<string?>("DeletedBy").HasMaxLength(256);
        builder.Property<DateTime?>("DeletedAt");
        builder.Property<string?>("IpAddress").HasMaxLength(45);
        builder.Property<string?>("Machine").HasMaxLength(256);

        builder.HasOne(x => x.Country)
               .WithMany()
               .HasForeignKey(x => x.CountryId)
               .HasConstraintName("fk_trade_agreements_countries")
               .OnDelete(DeleteBehavior.Restrict);
    }
}
```

### 16.2 Value Converters for Domain Types

```csharp
// Status enum stored as VARCHAR
builder.Property(x => x.Status)
       .HasConversion(
           v => v.ToString(),
           v => Enum.Parse<AgreementStatus>(v));
```

### 16.3 Owned Entities for Value Objects

Domain value objects (e.g., `Money`, `DateRange`) are mapped as owned entities to avoid a separate table when the value object is always loaded with the owner.

```csharp
builder.OwnsOne(x => x.Price, money =>
{
```

## 17. Query Optimization

### 17.1 EXPLAIN ANALYZE

Before merging any new Dapper query or EF Core LINQ expression that touches a table with more than 10,000 rows, run `EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)` on a representative dataset.

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT ta.id, ta.agreement_name, ta.status
FROM   public.trade_agreements ta
WHERE  ta.tenant_id = 'some-uuid'
  AND  ta.is_deleted = FALSE
ORDER BY ta.agreement_name
LIMIT 20 OFFSET 0;
```

Flag any query plan that shows `Seq Scan` on a table with more than 5,000 rows.

### 17.2 Slow Query Threshold

Queries taking more than 100ms in production are slow queries. Log them automatically via `pg_stat_statements` and a custom `NpgsqlDataSource` command filter. Investigate any query appearing in the slow log more than 10 times per hour.

### 17.3 N+1 Prevention

EF Core navigation properties must use `Include()` or `ThenInclude()` explicitly. Lazy loading is disabled globally. Any query that results in N+1 database round-trips must be rewritten using a JOIN-based Dapper query or an eager-loaded EF Core query.

## 18. Locking

### 18.1 SELECT FOR UPDATE SKIP LOCKED

For processing queues (e.g., pending calculation executions), use `SELECT FOR UPDATE SKIP LOCKED` to allow concurrent workers without deadlocks:

```sql
SELECT id, product_id, agreement_id
FROM   public.calculation_executions
WHERE  status = 'Pending'
  AND  tenant_id = @TenantId
ORDER BY executed_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

### 18.2 Avoid Table Locks

`LOCK TABLE` statements are forbidden in application code. They are permitted only in maintenance scripts during scheduled downtime windows, and only with explicit team lead approval.

Advisory locks (`pg_try_advisory_lock`) are acceptable for distributed mutex scenarios where optimistic concurrency is insufficient.

## 19. Concurrency

### 19.1 Optimistic Concurrency with xmin

PostgreSQL's `xmin` system column is the preferred optimistic concurrency token. It is updated automatically by PostgreSQL on every row update.

```csharp
// In IEntityTypeConfiguration
builder.UseXminAsConcurrencyToken();
```

### 19.2 Handling DbUpdateConcurrencyException

```csharp
try
{
    await _context.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    var entry = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync();
    if (dbValues is null)
        throw new DomainException("The record was deleted by another user.");

    // Refresh from DB and let the caller decide how to merge
    entry.OriginalValues.SetValues(dbValues);
    throw new ConcurrencyConflictException("The record was modified by another user. Refresh and retry.");
}
```

## 20. Backup and Recovery

### 20.1 Backup Schedule

| Type | Tool | Frequency | Retention |
|------|------|-----------|-----------|
| Logical backup | `pg_dump` (custom format) | Daily at 02:00 UTC | 30 days |
| Physical backup | `pg_basebackup` | Weekly Sunday 01:00 UTC | 4 weeks |
| WAL archiving | `archive_command` / pgBackRest | Continuous | 14 days |

WAL archiving enables Point-in-Time Recovery (PITR) to any second within the WAL retention window.

### 20.2 Restore Testing

| Activity | Frequency | Owner |
|---------|-----------|-------|
| Full restore drill to isolated environment | Monthly | DBA / DevOps |
| Verify row counts and referential integrity post-restore | Each restore drill | DBA |
| Document restore duration | Each restore drill | DBA |

**RTO target:** 4 hours for full database restore from physical backup.
**RPO target:** 5 minutes maximum data loss (WAL archiving interval).

Restore drill results are logged in the operations runbook. Any restore exceeding the RTO target triggers a backup strategy review.

## 21. Database Security

### 21.1 Database Roles

| Role | Permissions | Used by |
|------|-------------|---------|
| `app_user` | `SELECT`, `INSERT`, `UPDATE` on `public`, `rule_engine` schemas; `USAGE` on sequences | Application runtime |
| `read_only` | `SELECT` on all schemas | Reporting, Dapper read-side, analytics |
| `migration_user` | `CREATE`, `ALTER`, `DROP` on all schemas; owns migration objects | EF Core migration runner (CI/CD only) |
| `audit_writer` | `INSERT` on `audit` schema only | Audit trigger functions |

No application connection string uses a superuser or the `postgres` role. If a connection string contains `password=postgres` or `username=postgres`, it is a critical security defect.

### 21.2 Principle of Least Privilege

- `app_user` cannot `DELETE` rows (all deletes are soft; if a hard delete is ever required, it goes through a dedicated `DELETE`-privileged function reviewed by two developers).
- `app_user` cannot `TRUNCATE`, `DROP`, or `ALTER` any table.
- `read_only` cannot write to any table.
- Schema `audit` is write-accessible only through the `audit_writer` role bound to trigger functions — the application cannot directly `INSERT` into audit tables.

### 21.3 Connection Security

- All connections use TLS (`sslmode=require` minimum, `verify-full` in production).
- Passwords are stored in a secrets manager (Azure Key Vault or equivalent); no credentials in application configuration files or environment variable files committed to source control.
- Connection pool size is bounded: maximum 20 connections per application instance for `app_user`.
